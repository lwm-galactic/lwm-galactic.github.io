Redisearch

Docker 部署

1. 拉取 Redis Stack 镜像

Redis Stack 是 Redis 官方提供的一个集成镜像，包含了 Redis 及其多个模块（如 RediSearch、RedisJSON、RedisGraph 等）。

2. 启动 Redis Stack 容器

运行以下命令启动 Redis Stack 容器：

  • --name redis-stack：为容器指定名称（这里是 redis-stack）。
  • -p 6379:6379：将容器的 6379 端口映射到主机的 6379 端口。
  • -d：以后台模式运行容器。

概念学习

1. 索引（Index）

新概念

  • 索引的定义 ：
    • 在 Redis 中，数据以键值对的形式存储，没有内置的索引机制。
    • RediSearch 引入了“索引”这一概念，允许用户基于字段创建结构化的索引，支持全文搜索、数值范围查询和地理空间查询等操作。

作用

  • 索引是 RediSearch 的核心，它定义了文档的结构（字段及其类型），并为后续的搜索和查询提供支持。
  • 索引可以绑定到 Redis 的哈希（HASH）或 JSON 数据源上，并自动同步数据变化。

2. 文档（Document）

新概念

  • 文档的概念 ：
    • 在 Redis 中，数据以键值对的形式存在，每个键对应一个值（如字符串、列表、集合等）。
    • RediSearch 引入了“文档”的概念，类似于关系型数据库中的一行记录或 NoSQL 数据库中的文档。

作用

  • 每个文档由多个字段组成，字段可以有不同的类型（如文本、数值、标签等）。
  • 文档通过唯一的 ID 存储在 Redis 中（通常是哈希或 JSON 数据）。

3. 字段类型与属性

新概念

  • 字段类型 ：
    • Redis 本身没有字段类型的概念，所有数据都以字符串形式存储。
    • RediSearch 引入了多种字段类型，例如：
      • TEXT：用于全文搜索。
      • NUMERIC：用于数值范围查询。
      • TAG：用于精确匹配的标签。
      • GEO：用于地理空间查询。
  • 字段属性 ：
    • 每个字段可以附加属性，例如：
      • WEIGHT：设置字段的权重，影响搜索结果的相关性。
      • SORTABLE：允许对该字段进行排序。
      • NOSTEM：禁用词干提取。
      • PHONETIC：启用同音匹配。

4. 查询语言

新概念

  • 查询语法 ：
    • Redis 原生不支持复杂的查询语法。
    • RediSearch 提供了一种类布尔表达式的查询语言，支持多种条件组合。

作用

  • 允许用户执行全文搜索、布尔逻辑查询、范围查询、排序和分页等操作。

5. 自动补全与建议

新概念

  • 自动补全 ：
    • Redis 原生不支持自动补全功能。
    • RediSearch 提供了自动补全（Auto-complete）功能，适合实现搜索框的智能提示。

6. 地理空间查询

新概念

  • 地理字段 ：
    • Redis 原生支持简单的地理空间操作（如 GEOADD 和 GEORADIUS），但功能有限。
    • RediSearch 引入了专门的 GEO 字段类型，支持更复杂的地理空间查询。

作用

  • 可以根据地理位置坐标（经纬度）查找附近的点。

7. 聚合查询

新概念

  • 聚合操作 ：
    • Redis 原生不支持分组、统计等聚合操作。
    • RediSearch 提供了 FT.AGGREGATE 指令，支持类似 SQL 的聚合功能。

作用

  • 可以对搜索结果进行分组、计数、求和、平均等操作。

8. 高级搜索功能

新概念

  • 模糊匹配与同音匹配 ：
    • Redis 原生不支持模糊匹配或同音匹配。
    • RediSearch 提供了模糊匹配（Fuzzy Matching）和同音匹配（Phonetic Matching）功能。

作用

  • 提高搜索的灵活性和用户体验。

9. JSON 数据支持

新概念

  • JSON 数据源 ：
    • Redis 原生不支持 JSON 数据格式。
    • 结合 RedisJSON 模块，RediSearch 支持基于 JSON 数据的索引和查询。

作用

  • 允许直接从 JSON 数据中提取字段并创建索引。

指令学习

FT.CREATE

  • 作用

    FT.CREATE 是 RediSearch 模块中的一个核心指令，用于创建全文搜索索引。索引是 RediSearch 的基础结构，它定义了文档的字段及其类型，并为后续的搜索操作提供支持。

  • 使用方式

    参数说明：

    1. index_name ：
      • 索引的名称，必须是唯一的。
    2. ON data_type （可选，默认为 HASH）：
      • 指定索引的数据源类型。目前支持两种类型：
        • HASH：索引基于 Redis 哈希结构。
        • JSON：索引基于 RedisJSON 模块中的 JSON 数据（需要安装 RedisJSON 模块）。
    3. PREFIX count prefix ... （可选）：
      • 指定索引适用的键前缀。
      • count 表示前缀的数量，后面跟随具体的前缀字符串。
      • 例如：PREFIX 1 doc: 表示索引只适用于以 doc: 开头的键。
    4. SCHEMA ：
      • 定义索引的字段及其类型。
      • 每个字段由字段名、字段类型和可选属性组成。
    5. 字段类型与选项 ：
      • TEXT ：用于全文搜索的文本字段。
        • 可选属性：
          • WEIGHT <value>：设置字段的权重，默认为 1.0。
          • NOSTEM：禁用词干提取。
          • PHONETIC <matcher>：启用同音匹配。
      • NUMERIC ：用于数值范围查询的字段。
        • 可选属性：
          • SORTABLE：允许对该字段进行排序。
      • TAG ：用于精确匹配的标签字段。
        • 可选属性：
          • SEPARATOR <char>：指定标签分隔符，默认为逗号 ,。
      • GEO ：用于地理空间查询的字段。

数据持久化

RDB（Redis Database File）

RDB 是 Redis 的一种基于快照的持久化方式。它会将某一时刻 Redis 中的所有数据保存到一个二进制文件中（默认文件名为 dump.rdb）。当 Redis 重启时，可以通过加载这个文件来恢复数据。

工作原理

  • RDB 会在指定的时间间隔内，或者在满足某些条件时，触发一次数据快照操作。
  • 快照操作会将内存中的数据写入到磁盘上的 RDB 文件中。
  • Redis 在执行快照时会 fork 一个子进程，由子进程负责将数据写入磁盘，主进程继续处理客户端请求，因此对性能的影响较小。

优点

  1. 文件紧凑 ：RDB 文件是经过压缩的二进制文件，占用空间小，适合备份。
  2. 恢复速度快 ：加载 RDB 文件时，Redis 可以快速恢复大量数据。
  3. 对性能影响小 ：由于快照操作由子进程完成，主进程不会阻塞。

缺点

  1. 数据丢失风险 ：如果 Redis 宕机，可能会丢失最近一段时间的数据（最后一次快照之后的数据）。
  2. 大规模数据集时的性能问题 ：对于非常大的数据集，创建快照可能会消耗较多时间，并对系统资源造成压力。

AOF （Append-Only File）

AOF 是 Redis 的另一种持久化方式，它通过记录每个写操作命令来实现持久化。Redis 会将所有的写操作追加到一个日志文件中（默认文件名为 appendonly.aof），在 Redis 重启时，可以通过重新执行这些命令来恢复数据。

工作原理

  • 每次执行写操作时，Redis 会将该操作以文本形式追加到 AOF 文件中。
  • 当 Redis 启动时，会逐条读取并执行 AOF 文件中的命令，从而恢复数据。

  1. 数据完整性高 ：AOF 记录了每次写操作，因此可以最大限度地减少数据丢失。
  2. 可读性强 ：AOF 文件是以文本形式存储的，便于查看和手动修复。
  3. 灵活性高 ：可以通过配置 appendfsync 参数来权衡性能和安全性。

缺点

  1. 文件体积较大 ：AOF 文件通常比 RDB 文件大，因为它记录了所有写操作。
  2. 恢复速度较慢 ：由于需要逐条执行命令，AOF 恢复数据的速度通常比 RDB 慢。
  3. 可能存在冗余命令 ：如果写操作频繁，AOF 文件可能会包含大量重复或无用的命令。
