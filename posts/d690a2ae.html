向量化前文本预处理

为什么要对文本进行预处理

在进行文本向量化（如使用 BERT、Sentence-BERT、Word2Vec、TF-IDF 等模型）之前，对文本进行预处理 是非常关键的一步。这是因为原始文本通常包含很多“噪声”或不利于模型理解的内容，如果不经过清洗和标准化，会直接影响后续任务的效果。

✅ 一、为什么要对文本进行预处理？核心原因如下：

1. 去除噪声，提高语义准确性

  • 原始文本中可能包含无意义字符、特殊符号、HTML标签、乱码等。
  • 这些内容会对模型造成干扰，影响词向量的质量。

2. 统一格式，提升一致性

  • 不同来源的文本格式不一致（大小写、标点、空格、缩进等），会影响模型识别相似性。

🧼 二、常见的文本预处理步骤

步骤 描述 示例
1. 清洗文本 删除 HTML 标签、特殊符号、换行符、不可见字符等 <p>Hello</p>→Hello
2. 分句 / 分词 将文本切分为句子或词语（英文按空格，中文需分词） 我爱自然语言处理→["我", "爱", "自然语言", "处理"]
3. 统一大小写 英文转为小写 "This IS a Test"→"this is a test"
4. 去除数字 / 特殊字符 可选地移除纯数字或非语义字符 "Price: $100"→"Price"
5. 控制长度 截断或填充文本以适应模型最大长度限制 超过512token则截断

📈 三、预处理对下游任务的影响

任务 影响程度
文本分类 高：准确率依赖语义清晰度
相似度匹配 高：语义一致性决定结果质量
问答系统 高：上下文连贯性至关重要
摘要生成 中高：信息完整性依赖预处理效果
关键词提取 中：去噪和停用词过滤直接影响关键词准确性

✅ 总结一句话：

文本预处理是为了让模型“看得更清楚”，从而“理解得更准确”。它是高质量向量化和智能应用的基础。

文本处理的两个重要概念

🧩 一、chunksize：块大小（Chunk Size）

✅ 定义：

  • chunksize 是指每个文本切片（chunk）中包含的最大 token 数量 （不是字符数或句子数量）。
  • 在向量化（embedding）、检索增强生成（RAG）、上下文窗口等场景中非常关键。

🔒 限制因素：

  1. Embedding 模型最大输入长度
  • 不同 embedding 模型对输入长度有上限。例如：
    • all-MiniLM-L6-v2: 最大长度为 256 tokens
    • text-embedding-ada-002: 支持最多 8192 tokens
    • BGE-large: 支持 512 tokens

如果你的 chunk 超过模型支持的最大 token 数，则会被截断（truncate），导致信息丢失。

  1. LLM 的上下文窗口
  • LLM（如 GPT-3.5、GPT-4、Llama 等）也有最大上下文长度限制：
    • GPT-3.5-turbo: 4096 tokens
    • GPT-4: 8192 或 32768（取决于版本）
    • Llama2-70b: 4096 tokens

所以在 RAG 中，如果要将多个 chunks 拼接到 prompt 中，总 token 数不能超过这个限制。

🧠 建议设置：

Embedding Model Max Chunk Size
all-MiniLM-L6-v2 256
BERT-base 512
BGE-M3 8192
text-embedding-ada-002 8192

实际使用时建议预留一些 buffer，比如设置为模型最大长度的 80%。

🔁 二、overlap：重叠（Overlap）

✅ 定义：

  • overlap 表示相邻两个 chunk 之间的重叠 token 数量。
  • 目的是为了防止在切分时“切断”语义连贯的内容，保留上下文信息。

🎯 使用目的：

  1. 缓解语义断裂 ：避免一个完整句子/段落被切成两半。
  2. 提升召回率 ：在检索时，即使关键词出现在两个 chunk 的交界处，也能保证至少在一个 chunk 中出现完整的上下文。
  3. 鲁棒性增强 ：在问答系统中，有助于模型理解跨 chunk 的上下文。

⚖️ 设置建议：

chunksize overlap (建议值)
512 64 ~ 128
1024 128 ~ 256
2048 256 ~ 512

一般设置为 chunksize 的 10%~20% 即可。

📊 三、chunksize vs overlap 对比表

参数 含义 影响 推荐范围
chunksize 每个 chunk 的最大 token 数 控制信息密度、影响 embedding 效果、受限于模型长度 128 ~ 2048
overlap 相邻 chunk 之间的重叠 token 数 防止语义断裂、提高检索召回率 chunksize 的 10%~20%

文本切分策略

🧠 一、不同文本切分策略概览

策略 中文名称 特点 适用场景
Character TextSplitter 字符级切分器 按字符数固定长度切分 快速简单，适合格式不敏感的处理
RecursiveCharacter TextSplitter 递归字符切分器 按指定标点优先切分，避免断句 更智能的固定长度切分
Document Specific Splitting 文档专用切分 按文档结构（如标题、段落）切分 PDF、Markdown、Word 等结构化文档
Semantic Splitting 语义切分 按语义连贯性切分 高质量语义理解任务（摘要、问答等）

🛠️ 二、逐个详解 + 实现方式

1. Character TextSplitter（字符级切分器）

✅ 实现原理：
  • 将文本按固定字符数切分。
  • 不考虑句子边界或语义。
🔧 示例代码：
⚠️ 缺点：
  • 容易打断句子，导致语义断裂。
  • 对中文支持较差（一个词可能被拆成两个字）。

2. RecursiveCharacter TextSplitter（递归字符切分器）

✅ 实现原理：
  • 按照一组“分割符”进行优先尝试切分（如 ["\n\n", "\n", "。", ".", " "]）。
  • 如果找不到合适的分割符，则回退到字符切分。
🔧 示例流程：
  1. 先尝试用 \n\n 切分（段落级）
  2. 若不行，用 \n（换行）
  3. 再不行，用 。 或 .（句号）
  4. 最后用空格、字符切分
⚠️ 优点：
  • 避免在句子中间切断。
  • 保持语义完整性优于字符切分。

3. Document Specific Splitting（文档专用切分）

✅ 实现原理：
  • 根据文档类型（PDF、Markdown、HTML、Word）提取结构信息。
  • 按照标题、段落、列表等结构进行切分。
🔧 示例流程：
  1. 使用 OCR 或解析器读取 PDF/HTML。
  2. 提取标题层级（如 H1, H2）、段落标签。
  3. 每个章节或子标题作为一个 Chunk。
📦 代码示例：
  • PDF：pdfplumber, PyPDF2
  • Markdown：markdownify, mistune
  • HTML：BeautifulSoup
⚠️ 优点：
  • 结构清晰，适合知识库构建。
  • 保留逻辑层次。

4. Semantic Splitting（语义切分） - 滑动窗口法

✅ 实现原理：
  • 按句子初步切分。
  • 使用滑动窗口遍历句子组合。
  • 通过语义模型判断是否应合并为一个语义单元。
🧩 步骤详解（以你提供的六个语义单元为例）
Step 1: 预处理 → 句子切分
Step 2: 构建滑动窗口（窗口大小=2，步长=1）
Step 3: 语义分析（使用 BERT 计算相似度）
Step 4: 输出结果（语义单元）

🧪 三、总结对比表

策略 是否保留语义 是否保留结构 速度 精度 应用建议
Character ❌ 否 ❌ 否 ⏱️ 快 ⬇️ 低 快速预处理
RecursiveCharacter ✅ 是 ❌ 否 ⏱️ 较快 ⬆️ 中 通用文本处理
Document Specific ✅ 是 ✅ 是 ⏳ 慢 ⬆️ 中高 结构化文档处理
Semantic Splitting ✅ 是 ❌ 否 ⏳ 很慢 ⬆⬆ 高 高质量语义任务

📝 四、如何选择？

场景 推荐策略
快速索引构建 RecursiveCharacter
PDF/Word 文档处理 Document Specific
QA / Summarization / RAG Semantic Splitting
简单粗暴处理 Character
